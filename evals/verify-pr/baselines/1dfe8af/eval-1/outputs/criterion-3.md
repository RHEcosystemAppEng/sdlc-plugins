## Criterion 3: `GET /api/v2/package?license=INVALID-999` returns 400 Bad Request with an error message

### Verdict: PASS

### Evidence

**Endpoint layer (`modules/fundamental/src/package/endpoints/list.rs`):**

The `validate_license_param` function validates each license identifier by parsing it as an SPDX expression:

```rust
Expression::parse(id).map_err(|_| {
    AppError::BadRequest(format!("Invalid SPDX license identifier: {}", id))
})?;
```

`INVALID-999` is not a recognized SPDX license identifier, so `Expression::parse("INVALID-999")` will return an `Err`. This is mapped to `AppError::BadRequest` with the message `"Invalid SPDX license identifier: INVALID-999"`. The `?` operator propagates this error, causing the handler to return early with a 400 response.

The handler invokes validation before the database query:

```rust
let license_filter = match &params.license {
    Some(license) => Some(validate_license_param(license)?),
    None => None,
};
```

The `?` at the end of `validate_license_param(license)?` ensures the `AppError::BadRequest` is returned as the handler's error response, which (per the repository's `AppError` implementation in `common/src/error.rs`) translates to a 400 HTTP status code.

The test `test_list_packages_invalid_license_returns_400` queries `?license=INVALID-999` and asserts:
- Response status is `StatusCode::BAD_REQUEST` (400)

This directly validates the criterion. The error message is generated by the format string `"Invalid SPDX license identifier: {}"`, satisfying the "with an error message" requirement.
