# Step 4 -- Root Cause Analysis: ACME-510

## Root Cause

The `/api/v2/advisories` endpoint uses separate code paths for unfiltered and
date-range-filtered requests. The unfiltered path correctly computes the total
count of matching records and attaches `X-Total-Count` and `Link` pagination
headers to the response. The date-range-filtered path (triggered by `publishedAfter`
and `publishedBefore` query parameters) executes the filtered query and returns
the correct result set, but it does not invoke the total count computation or
the pagination header attachment logic.

This is a missing-integration defect: the filtered query path was implemented
without wiring it through the same pagination header pipeline that the unfiltered
path uses.

## Root Cause Breakdown

### What is broken

The date-range-filtered query path for `GET /api/v2/advisories` does not compute
the total count of matching records and does not attach `X-Total-Count` or `Link`
pagination headers to the HTTP response.

### Why it is broken

The filtered code path was implemented as a separate branch that constructs and
executes the filtered database query but bypasses (or was never connected to) the
shared pagination utility that computes total count and generates pagination headers.
The unfiltered path already calls this utility correctly, but the filtered path
returns results directly without invoking it.

### Where it is broken

- **Endpoint handler** (e.g., `src/api/v2/advisories.rs`): the conditional branch
  for date-range filtering does not call the pagination header utility after
  executing the filtered query.
- **Query builder / repository layer**: the filtered query method may not support
  returning a total count alongside the result set, or the count query does not
  apply the date range filters.
- **Pagination utilities** (e.g., `src/api/pagination.rs`): the utility itself is
  likely correct (since unfiltered requests work), but it is not being called from
  the filtered path.

### How to verify the fix

A reproducer test should:

1. Send `GET /api/v2/advisories?publishedAfter=2025-01-01&publishedBefore=2025-06-30&limit=10`
   to the service (or call the handler directly in an integration test).
2. Assert that the response includes the `X-Total-Count` header with the correct
   total number of matching advisories.
3. Assert that the response includes the `Link` header with `rel="next"` when more
   pages exist beyond the requested limit.
4. Assert that the response body still contains the correct filtered advisories
   (no regression in result content).
