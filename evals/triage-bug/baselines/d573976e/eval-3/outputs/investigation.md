## Step 2 -- Code-Path Tracing

This bug involves skill/documentation behavior in the sdlc-workflow plugin and cannot be directly reproduced via CLI commands. Code-path tracing was performed instead.

### Trace 1: Convention Reference Formatting

Entry point: `/plan-feature` skill invocation.

The convention reference formatting occurs when the plan-feature skill reads a repository's CONVENTIONS.md and generates Implementation Notes for task descriptions. The formatter in `shared/convention-utils.md` is responsible for converting CONVENTIONS.md section headings into reference tokens for use in task descriptions.

**Divergence point**: The convention formatter applies a kebab-case transformation to section headings (e.g., `## Migration Patterns` becomes `migration-patterns`) when generating section reference tokens. The correct behavior should preserve the original heading text as-is (e.g., `Migration Patterns`), producing references like `Migration Patterns` rather than `migration-patterns`.

### Trace 2: Task Issue Type Selection

Entry point: `/plan-feature` skill invocation, specifically the task creation step.

The task creation logic in `plan-feature/SKILL.md` Step 6a determines which Jira issue type to use when creating tasks. The logic reads the issue type ID from the project's Jira Configuration.

**Divergence point**: The task creation logic reads the `Feature issue type ID` (10142) from Jira Configuration instead of using the correct Task issue type ID. When a project has a custom issue type scheme where Task has a different ID (e.g., 10050), the logic should be looking up the Task issue type ID, not the Feature issue type ID.

### Affected Files and Modules

**Root Cause 1 -- Convention Reference Formatter**:
- **File**: `shared/convention-utils.md`
- **Defect**: The convention reference formatting logic lowercases and kebab-cases section headings from CONVENTIONS.md instead of preserving the original heading text.
- **Impact**: All task descriptions generated by plan-feature (and potentially other skills that reference conventions) will have malformed convention references in their Implementation Notes.

**Root Cause 2 -- Task Issue Type Selection**:
- **File**: `plan-feature/SKILL.md`, Step 6a (task creation)
- **Defect**: The task creation step reads `Feature issue type ID` from Jira Configuration when it should read the Task issue type ID. Under custom issue type schemes, this causes tasks to be created with the wrong issue type.
- **Impact**: All tasks created by plan-feature in projects with custom issue type schemes will have the wrong issue type (Feature instead of Task).

### Independence Assessment

These two root causes are **independent**:

1. They reside in **different modules**: `shared/convention-utils.md` (shared utility) vs. `plan-feature/SKILL.md` (skill-specific logic).
2. They affect **different code paths**: one is in the text formatting pipeline for task descriptions, the other is in the Jira API call for issue creation.
3. They can be **fixed independently**: fixing the convention formatter does not affect issue type selection, and vice versa.
4. They have **different scopes of impact**: the convention formatter bug affects all skills that use convention references, while the issue type bug is specific to plan-feature's task creation step.

## Step 4 -- Root Cause Analysis

### Root Cause 1: Convention Reference Formatter (shared/convention-utils.md)

- **What is broken**: The convention reference formatter transforms CONVENTIONS.md section headings into kebab-case tokens (e.g., `## Migration Patterns` becomes `migration-patterns`) instead of preserving the original heading text.
- **Why it is broken**: The formatter applies an unnecessary normalization step (lowercase + kebab-case) to heading text when generating section reference identifiers. This normalization is appropriate for URL anchors but not for human-readable convention references in task descriptions.
- **Where it is broken**: `shared/convention-utils.md` -- the heading-to-reference conversion logic.
- **How to verify the fix**: A reproducer test should:
  1. Provide a CONVENTIONS.md with a section heading `## Migration Patterns`
  2. Invoke the convention reference formatter
  3. Assert that the output reference is `Migration Patterns` (title case, matching the heading), not `migration-patterns` (kebab-case)

### Root Cause 2: Task Issue Type Selection (plan-feature/SKILL.md Step 6a)

- **What is broken**: The task creation logic in plan-feature reads the Feature issue type ID (10142) from Jira Configuration and uses it as the issue type for created tasks, instead of using the Task issue type ID.
- **Why it is broken**: The code references the wrong configuration field. It reads `Feature issue type ID` when it should read or derive the Task issue type ID from the project's issue type scheme.
- **Where it is broken**: `plan-feature/SKILL.md`, Step 6a -- the issue type parameter in the `jira.create_issue` call.
- **How to verify the fix**: A reproducer test should:
  1. Configure a project with a custom issue type scheme where Task has ID 10050
  2. Run plan-feature to create a task
  3. Assert that the created issue has issue type ID 10050 (Task), not 10142 (Feature)

### Conclusion

ACME-502 describes **two independent bugs** with distinct root causes in different modules. Each root cause requires a separate fix in a separate code path. The Decomposition Guard (Step 6) is triggered.
