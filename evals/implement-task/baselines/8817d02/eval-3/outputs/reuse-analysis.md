# Reuse Analysis for TC-9203: Add package license filter to list endpoint

This document details how each of the three Reuse Candidates identified in the task description would be applied during implementation, avoiding any duplication of existing logic.

## Reuse Candidate 1: `common/src/db/query.rs::apply_filter`

**What it provides:** The `apply_filter` function handles comma-separated multi-value query parameter parsing and SQL `IN` clause generation. It takes a raw query parameter string (e.g., `"MIT,Apache-2.0"`), splits it on commas, and produces the appropriate SeaORM filter condition -- either an equality check for a single value or an `IN` clause for multiple values.

**How it would be reused:** In `modules/fundamental/src/package/service/mod.rs`, the `PackageService::list` method would call `apply_filter` directly with the `license` query parameter value and the `package_license` column reference. This handles all parsing and SQL generation for the license filter.

**What this avoids:** Without reusing `apply_filter`, the implementation would need to:
- Manually split the comma-separated string (e.g., `license.split(',')`)
- Handle the single-value vs. multi-value distinction
- Construct SeaORM `ColumnTrait::is_in()` or `ColumnTrait::eq()` conditions manually
- Handle edge cases like empty strings or trailing commas

All of this is already implemented and tested in `apply_filter`. Reusing it directly eliminates duplicate parsing logic and ensures consistent behavior with other filters across the application.

## Reuse Candidate 2: `modules/fundamental/src/advisory/endpoints/list.rs` (severity filter pattern)

**What it provides:** The advisory list endpoint implements a structurally identical filter pattern -- an optional `severity` query parameter on a list endpoint. It demonstrates:
- How to declare an optional filter field in the Axum Query struct (e.g., `severity: Option<String>`)
- How to extract the filter value from the deserialized query parameters
- How to pass the filter value to the corresponding service method
- How to validate the parameter and return 400 Bad Request for invalid values
- The overall handler structure for a filtered list endpoint

**How it would be reused:** The advisory severity filter serves as the structural template for the package license filter. Specifically:

1. **Query struct pattern:** In `modules/fundamental/src/package/endpoints/list.rs`, add a `license: Option<String>` field to the existing Query struct, following the exact same pattern as the `severity: Option<String>` field in the advisory endpoint's Query struct.

2. **Handler flow:** Follow the same control flow as the advisory list handler:
   - Extract the optional filter from the query parameters
   - Pass it to the service layer
   - Return the filtered `PaginatedResults<T>` response

3. **Validation pattern:** Follow the same validation approach used for the severity filter to validate license parameter values and return `AppError`-based 400 responses for invalid inputs.

**What this avoids:** Without following this pattern, the implementation might:
- Invent a different query parameter extraction approach
- Structure the handler differently from sibling endpoints
- Use a different error handling pattern for validation failures

By following the advisory severity filter pattern exactly, the package license filter maintains consistency with the established codebase conventions.

## Reuse Candidate 3: `entity/src/package_license.rs` (package-license join entity)

**What it provides:** The `package_license` SeaORM entity defines the mapping between packages and their licenses. It represents the `package_license` database table (a join/association table) and includes:
- The entity struct and column definitions
- SeaORM `Relation` definitions connecting packages to licenses
- The `Model` struct with fields for the package ID, license identifier, and any other join-table columns

**How it would be reused:** In `modules/fundamental/src/package/service/mod.rs`, when the `license` filter is present, the query would use the `package_license` entity to perform a JOIN:

1. **JOIN construction:** Use SeaORM's `join()` method with `JoinType::InnerJoin` on the `package_license::Entity` to connect the `package` table to the `package_license` table via their defined relation.

2. **Filter application:** After joining, apply the filter condition (generated by `apply_filter` from Reuse Candidate #1) to the `package_license` table's license identifier column (e.g., `package_license::Column::LicenseId`).

3. **Conditional join:** Only add the JOIN when the `license` parameter is provided. When `license` is `None`, the query skips the join entirely, preserving the original unfiltered behavior.

**What this avoids:** Without reusing the existing `package_license` entity, the implementation might:
- Write raw SQL `JOIN` statements instead of using SeaORM's type-safe join API
- Duplicate the table/column definitions already captured in the entity
- Miss the existing relation definitions, potentially joining on incorrect columns
- Introduce inconsistencies with how other parts of the codebase query the same join table

By using the existing entity, the implementation stays within SeaORM's type-safe query system and leverages the relation definitions that are already defined and tested.
