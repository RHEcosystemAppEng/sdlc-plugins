# Reuse Analysis: TC-9203 — Add package license filter to list endpoint

This document details how each Reuse Candidate listed in the task description is used
in the implementation, following the SKILL.md directive: "Reuse first: Before writing
new logic, check whether the Implementation Notes list reusable code. If they do, use
or extend the existing code."

## Reuse Candidate 1: `common/src/db/query.rs::apply_filter`

**Source:** `common/src/db/query.rs`

**What it provides:** Handles comma-separated multi-value query parameter parsing and
SQL IN clause generation. Given a raw string like `"MIT,Apache-2.0"`, it splits on
commas, trims whitespace, and produces the appropriate filter condition for use in
SeaORM queries.

**How it is reused:** Called directly in `modules/fundamental/src/package/service/mod.rs`
within the `PackageService::list()` method. When the `license` query parameter is
present, the raw `Option<String>` value is passed to `apply_filter`, which handles all
parsing and filter construction. No new parsing or splitting logic is written — the
existing function covers the full requirement for both single-value (`"MIT"`) and
multi-value (`"MIT,Apache-2.0"`) cases.

**Why no new code is needed:** The `apply_filter` function already implements exactly the
comma-separated-to-IN-clause transformation required by this task. Writing a new parser
or filter builder would duplicate this functionality and violate constraint 5.4 (no
duplication).

**Modification required:** None. `apply_filter` is used as-is with no changes to its
signature or implementation.

## Reuse Candidate 2: `modules/fundamental/src/advisory/endpoints/list.rs` (severity filter pattern)

**Source:** `modules/fundamental/src/advisory/endpoints/list.rs`

**What it provides:** A structurally complete reference implementation for adding an
optional filter query parameter to a list endpoint. The advisory list endpoint supports
a `severity` query parameter using:
1. An `Option<String>` field in the query parameter struct
2. Extraction in the handler and pass-through to the service layer
3. Delegation to `apply_filter` in the service for actual filtering

**How it is reused:** The entire structural pattern is replicated in the package list
endpoint:

- **Query struct field:** The `license: Option<String>` field in the package endpoint's
  query struct follows the same declaration pattern as `severity: Option<String>` in the
  advisory query struct.
- **Handler logic:** The package handler extracts `query.license` and passes it to
  `PackageService::list()` in the same way the advisory handler passes `query.severity`
  to `AdvisoryService::list()`.
- **Validation:** Input validation for the `license` parameter follows the same error
  handling pattern used for `severity` validation in the advisory endpoint.

**Why this pattern is followed instead of inventing a new one:** The advisory severity
filter is structurally identical to the license filter needed here. Using the same pattern
ensures consistency across the codebase and aligns with the convention conformance
analysis principle (match sibling file patterns rather than inventing new approaches).

**Modification required:** None to the advisory endpoint itself. The pattern is used as a
template for the package endpoint changes.

## Reuse Candidate 3: `entity/src/package_license.rs` (package_license entity)

**Source:** `entity/src/package_license.rs`

**What it provides:** The SeaORM entity definition for the `package_license` table, which
maps packages to their declared licenses. This includes:
- The entity struct and column definitions
- Relation definitions linking `package_license` to the `package` table
- The license column that holds the SPDX identifier

**How it is reused:** Used in `modules/fundamental/src/package/service/mod.rs` to
construct the JOIN between the `package` table and the `package_license` table when
filtering by license. Specifically:

- The entity's defined relations are used via SeaORM's `join()` or `find_also_related()`
  API to add a JOIN clause to the package list query.
- The entity's `Column::License` (or equivalent column enum variant) is used as the
  target column for the filter condition generated by `apply_filter`.

**Why no raw SQL is written:** The `package_license` entity already defines the table
schema and its relations to the `package` table. Using SeaORM's relation-based join
through this entity is both safer (compile-time checked) and consistent with how the
rest of the codebase handles entity relationships. Writing raw SQL JOIN clauses would
bypass SeaORM's type system and diverge from established patterns.

**Modification required:** None. The entity is used as-is — no new columns, relations,
or methods are added.
