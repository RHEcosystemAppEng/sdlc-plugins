## 3.2 -- Affects Versions Comparison and Correction

### Stream Scope

This issue has stream suffix `[rhtpa-2.2]` and is scoped to the **2.2.x** stream. Only 2.2.x versions are included in the Affects Versions correction. The 2.1.x versions (also affected per version impact table) belong to a companion/sibling issue for that stream.

### Current vs Proposed

**Current Affects Versions (PSIRT-assigned):** `[RHTPA 2.0.0]`

**PSIRT version is wrong**: RHTPA 2.0.0 does not exist in the configured Version Streams (no 2.0.x stream). PSIRT assigned a non-existent version.

**Proposed Affects Versions (from lock file analysis, scoped to 2.2.x stream):** `[RHTPA 2.2.0, RHTPA 2.2.1, RHTPA 2.2.2]`

These are the 2.2.x versions where openssl-libs is below the fix threshold (3.0.7-28.el9_4):
- RHTPA 2.2.0: openssl-libs 3.0.7-25.el9_3 -- affected
- RHTPA 2.2.1: openssl-libs 3.0.7-27.el9_4 -- affected
- RHTPA 2.2.2: retag of 2.2.1 -- affected (same as 2.2.1)

Versions 2.2.3 and 2.2.4 are NOT affected (ship 3.0.7-28.el9_4, the fix version) and are excluded.

### Correction Details

```
Current: [RHTPA 2.0.0] -> Proposed: [RHTPA 2.2.0, RHTPA 2.2.1, RHTPA 2.2.2]
```

**Rationale**: PSIRT assigned RHTPA 2.0.0, which does not match any configured version stream. Lock file analysis at pinned commits from security-matrix.md shows that versions 2.2.0, 2.2.1, and 2.2.2 in the 2.2.x stream ship vulnerable openssl-libs. Versions 2.2.3+ ship the fixed version (3.0.7-28.el9_4).
