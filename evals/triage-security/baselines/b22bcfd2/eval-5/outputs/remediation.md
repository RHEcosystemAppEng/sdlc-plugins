# Step 8 -- Remediation

## Triage Outcome: Case A -- Affected (create remediation task)

The version impact analysis shows that versions 2.2.0, 2.2.1, and 2.2.2 in the 2.2.x stream ship a vulnerable version of openssl-libs (below 3.0.7-28.el9_4). Remediation is required.

## Ecosystem: RPM (System Package)

Since openssl-libs is an RPM system package (not a source dependency like Cargo or npm), the remediation follows the **single-task path**. Only one remediation task is created in the Konflux release repo. There is no upstream backport + downstream propagation split -- RPM package fixes are applied directly in the Konflux release repo.

## Remediation Task: Explicit Install Fix

The dependency chain analysis (Step 2.3.5) classified openssl-libs as an **explicit install** based on its presence in rpms.lock.yaml. The remediation updates the package spec in the lock file.

After task creation, the description digest protocol is followed:

```

task_desc = jira.get_issue(<task-key>, fields=["description"])

python3 scripts/sha256-digest.py /tmp/task-desc.md
```

The digest is computed from the **re-fetched** description (via Jira API after `create_issue`), not from the description string passed to `create_issue`. This ensures the digest matches what Jira actually stored.

## Key Decisions

1. **Single task, not two**: RPM is a system package ecosystem. The fix is applied directly in the Konflux release repo (rpms.lock.yaml / rpms.in.yaml). No upstream backport task is needed because there is no upstream source repository branch for RPM packages.

2. **Explicit install template**: The dependency chain analysis confirmed openssl-libs is present in rpms.lock.yaml (explicit install), so the explicit install remediation template is used rather than the base image template.

3. **No Coordination Guidance**: The Source Repositories table lacks a Deployment Context column, so coordination guidance is omitted entirely per backward compatibility rules.
