## Architecture Summary

The implementation optimizes the existing search module (`modules/search/`) in three incremental stages:

1. **Performance foundation** (Task 1): Adds PostgreSQL full-text search infrastructure — tsvector columns/expressions and GIN indexes — via a new database migration. Updates the SearchService to use tsvector/tsquery-based queries instead of pattern matching.

2. **Relevance ranking** (Task 2): Builds on the full-text search infrastructure to add ts_rank-based relevance scoring. Exposes an optional `sort_by` query parameter and includes relevance scores in response items.

3. **Filtering** (Task 3): Extends the shared query builder helpers in `common/src/db/query.rs` to support entity type, date range, and entity-specific filters (severity for advisories, license for packages). Filters are additive query parameters on the existing search endpoint.

All changes are additive and backward-compatible. The existing `GET /api/v2/search` endpoint response shape is preserved; new fields and parameters are optional.
