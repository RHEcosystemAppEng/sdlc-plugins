## Architecture Summary

The plan replaces the `advisory_status` lookup table with a PostgreSQL enum column (`advisory_status_enum`) on the `advisory` table. The migration is structured as a feature-branch workflow with 6 intermediate implementation tasks bracketed by create-branch and merge-branch bookend tasks:

1. A reversible database migration creates the enum type, adds/backfills the column, and drops the lookup table
2. SeaORM entity definitions are updated to reflect the new schema (enum field replaces FK)
3. The advisory service layer and endpoints are updated to query the enum column directly
4. The ingestion pipeline is updated to write enum values instead of lookup table rows
5. Integration tests are updated for the new schema
6. Internal architecture documentation is updated
