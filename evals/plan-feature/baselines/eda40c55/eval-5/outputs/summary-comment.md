## Architecture Summary
This feature replaces the `advisory_status` lookup table with a PostgreSQL `advisory_status_enum` type and a direct `status` column on the `advisory` table. The migration handles enum type creation, column addition, data backfill, FK column removal, and table drop in a single atomic, reversible migration. The SeaORM entity definitions, advisory service layer, REST endpoints, ingestion pipeline, and integration tests are all updated to work with the new enum column. The API contract is unchanged — status is still returned as a string in API responses.

Feature-branch mode is used because the migration, entity changes, service updates, and ingestion changes are tightly coupled: merging any subset to `main` would leave the codebase in an inconsistent state.
