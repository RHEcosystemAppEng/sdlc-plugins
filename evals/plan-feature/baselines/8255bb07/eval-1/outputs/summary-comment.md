## Architecture Summary

The feature adds a server-side advisory severity aggregation endpoint to reduce frontend round-trips for severity breakdown data. The implementation follows the existing module pattern (`model/ + service/ + endpoints/`):

1. **Data layer**: A new `AdvisorySeveritySummary` model struct and a severity aggregation method on `SbomService` that queries the existing `sbom_advisory` join table with `GROUP BY` on the severity field, deduplicating by advisory ID.
2. **API layer**: A new `GET /api/v2/sbom/{id}/advisory-summary` endpoint registered in the SBOM module's route configuration, with 5-minute tower-http response caching and an optional `?threshold` query parameter for severity filtering.
3. **Ingestion hook**: Cache invalidation in the advisory ingestion pipeline (`modules/ingestor/src/graph/advisory/mod.rs`) to ensure freshness when new advisories are linked to SBOMs.
4. **Cross-cutting validation**: Smoke tests verify endpoint correctness and backward compatibility; performance benchmarks verify the p95 < 200ms latency requirement.

No new database tables are required — the feature uses existing entity tables (`sbom`, `advisory`, `sbom_advisory`).
